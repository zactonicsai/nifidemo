# Ansible Step-by-Step Tutorial

A practical, hands-on Ansible tutorial built around your `docker-compose.yml` lab. You already have a working controller (`ansible-controller`) and two target nodes (`target1`, `target2`) on a Docker bridge network called `lab-net`. This tutorial walks you from "containers are up" to "I'm writing real playbooks with roles, variables, and handlers."

## 1. What you already have

Your Docker Compose file describes three containers on a single bridge network:

| Container            | Role             | OS           | Notable ports                    |
|----------------------|------------------|--------------|----------------------------------|
| `ansible-controller` | Runs Ansible     | Ubuntu 24.04 | (none exposed to host)           |
| `target1`            | Managed node     | Ubuntu 24.04 | `8081 → 80`, sshd on 22 (internal) |
| `target2`            | Managed node     | Ubuntu 24.04 | `8082 → 80`, sshd on 22 (internal) |

A few things baked in by the Dockerfiles that matter for the tutorial:

- The **controller** has Ansible installed via `pipx`, plus `openssh-client`, `sshpass`, `ping`, and editors. The host directory `./ansible` is bind-mounted to `/ansible` inside the controller — that's your working directory, and anything you create there persists on your host.
- Each **target** runs `sshd` on port 22 (only reachable from the `lab-net` Docker network, not from your host), has `python3` and `python3-apt` installed (so Ansible's `apt` module works), has `nginx` pre-installed, and accepts SSH login as `root` with the password `root`. Targets are reachable from the controller by hostname: `target1` and `target2`.

That last point is the key insight: because Docker Compose puts all three containers on `lab-net`, the controller can `ssh root@target1` directly. No IP juggling needed.

## 2. Starting the lab

From the directory containing your `docker-compose.yml`, on your host:

```bash
docker compose up -d --build
```

`--build` ensures images are built from the Dockerfiles the first time (you can drop it on later runs). `-d` runs in the background.

Verify everything is healthy:

```bash
docker compose ps
```

You should see all three containers running, with `target1` and `target2` showing `(healthy)`. The healthcheck just probes TCP port 22 inside each target, so "healthy" means sshd is actually accepting connections — which is why `ansible-controller` is configured to wait for it via `depends_on: condition: service_healthy`.

If a target shows `(unhealthy)` or `(starting)`, give it ~10 seconds and re-check; sshd needs a moment after `ssh-keygen -A` on first boot.

## 3. Getting a shell on the controller

All Ansible commands run *from* the controller. Open a shell into it:

```bash
docker exec -it ansible-controller bash
```

You'll land in `/ansible` (the working directory set in the Dockerfile). This directory is bind-mounted from `./ansible` on your host, so files you create here show up on your host too — handy for editing in your normal editor.

Confirm Ansible is installed:

```bash
ansible --version
```

You should see something like `ansible [core 2.x.x]` plus the Python interpreter path. If this fails, the `pipx` install didn't put `~/.local/bin` on PATH — the Dockerfile sets `ENV PATH="/root/.local/bin:${PATH}"`, so it should just work.

## 4. Bootstrapping SSH access

Ansible talks to managed nodes over SSH. Two things need to happen before that works cleanly:

1. The controller has to **trust** each target's host key (otherwise SSH prompts interactively and Ansible hangs).
2. We need credentials that work. Two options: password auth (`sshpass`, which is already installed) or key-based auth (cleaner, recommended).

### 4a. Accept host keys

Still inside the controller shell:

```bash
ssh-keyscan -H target1 target2 >> ~/.ssh/known_hosts
```

If `~/.ssh` doesn't exist yet, create it first: `mkdir -p ~/.ssh && chmod 700 ~/.ssh`.

`ssh-keyscan` grabs each host's public key and `-H` hashes the hostnames in the file. This is what lets SSH connect non-interactively.

### 4b. Set up SSH keys (recommended)

Generate a keypair on the controller if you don't have one:

```bash
ssh-keygen -t ed25519 -N "" -f ~/.ssh/id_ed25519
```

`-N ""` means no passphrase (fine for a lab). Then copy the public key to both targets. The targets accept root login with password `root`, so `ssh-copy-id` works:

```bash
sshpass -p root ssh-copy-id -o StrictHostKeyChecking=no root@target1
sshpass -p root ssh-copy-id -o StrictHostKeyChecking=no root@target2
```

Test it — these should each return the hostname **without** asking for a password:

```bash
ssh root@target1 hostname
ssh root@target2 hostname
```

If you'd rather skip keys entirely and just use the password, you can; we'll show that style in the inventory too.

## 5. Your first inventory

The **inventory** tells Ansible what hosts exist and how to reach them. It can be INI-style or YAML. Create `/ansible/inventory.ini`:

```ini
[web]
target1
target2

[web:vars]
ansible_user=root
ansible_ssh_private_key_file=~/.ssh/id_ed25519
ansible_python_interpreter=/usr/bin/python3
```

What each piece does:

- `[web]` is a **group** named `web` containing two hosts.
- `[web:vars]` sets variables that apply to every host in the group: log in as `root`, use the SSH key we just created, and use the system Python on the target (which has `python3-apt` installed so the `apt` module works).

If you skipped the key setup, swap the key line for password auth:

```ini
[web:vars]
ansible_user=root
ansible_password=root
ansible_python_interpreter=/usr/bin/python3
```

Verify the inventory parses:

```bash
ansible-inventory -i inventory.ini --list
```

You'll get a JSON dump of groups and host vars. If it's empty or errors, the file is malformed.

> **About `ansible.cfg`:** you can avoid passing `-i inventory.ini` to every command by creating `/ansible/ansible.cfg`:
> ```ini
> [defaults]
> inventory = ./inventory.ini
> host_key_checking = False
> stdout_callback = yaml
> ```
> `host_key_checking = False` is fine for a lab; in production you'd manage `known_hosts` properly instead. `stdout_callback = yaml` makes output much more readable. With this file in place, you can drop `-i inventory.ini` from every example below.

## 6. Your first ad-hoc commands

Ad-hoc commands run a single module against your inventory, no playbook required. Great for one-offs and for sanity-checking.

**Ping every host** (this uses Ansible's `ping` module, which actually runs Python on the remote — it's not ICMP):

```bash
ansible -i inventory.ini web -m ping
```

Expected: each host returns `pong`.

**Run a shell command** on the whole group:

```bash
ansible -i inventory.ini web -m command -a "uptime"
```

**Gather full facts** (lots of output — every fact Ansible can discover about the host):

```bash
ansible -i inventory.ini target1 -m setup
```

**Install a package** (note `--become`, which elevates to root via sudo; we're already root here so it's redundant, but it's the habit you want):

```bash
ansible -i inventory.ini web -m apt -a "name=htop state=present update_cache=yes" --become
```

The pattern after `ansible` (`web`, `target1`, `all`) is a **host pattern**. You can combine groups, exclude hosts, use wildcards — `web:!target2` means "web group except target2".

## 7. Your first playbook

A **playbook** is a YAML file describing a series of tasks (and other things) to run against hosts. Create `/ansible/01-webserver.yml`:

```yaml
---
- name: Configure a basic nginx web server
  hosts: web
  become: true
  gather_facts: true

  tasks:
    - name: Ensure nginx is installed
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Ensure nginx is running and enabled
      ansible.builtin.service:
        name: nginx
        state: started

    - name: Drop a custom index page
      ansible.builtin.copy:
        dest: /var/www/html/index.html
        content: "<h1>Hello from {{ inventory_hostname }}</h1>\n"
        owner: root
        group: root
        mode: "0644"
```

Run it:

```bash
ansible-playbook -i inventory.ini 01-webserver.yml
```

A few things to notice in the output:

- Each task reports `ok`, `changed`, or `failed` per host.
- The first run shows `changed` for the copy task; run it again and it'll say `ok` — Ansible is **idempotent**: it only acts when state differs from what you declared.
- The recap at the bottom summarizes counts per host.

Now check the result. From your **host machine** (outside the containers):

```bash
curl http://localhost:8081   # target1 → "Hello from target1"
curl http://localhost:8082   # target2 → "Hello from target2"
```

This works because Compose maps `target1:80 → host:8081` and `target2:80 → host:8082`.

### Why we don't `service: enabled: yes` on these containers

In a real VM you'd add `enabled: yes` so nginx starts on boot. These containers run `sshd` as PID 1, not systemd, so there's no init system to "enable" against — `state: started` is enough for the lab. Worth knowing because some Ansible tutorials will tell you to add `enabled` and it'll fail on container hosts.

## 8. Variables, facts, and templates

Hardcoding `<h1>Hello from ...</h1>` directly into a task is fine for a demo but doesn't scale. Real playbooks separate **data** (variables) from **logic** (tasks) and use **templates** for any non-trivial file content.

### Variables

Create `/ansible/group_vars/web.yml` (Ansible auto-loads `group_vars/<groupname>.yml` for that group):

```yaml
---
site_title: "Ansible Lab"
site_admin_email: "admin@example.test"
nginx_worker_processes: 2
```

### Templates

Ansible uses Jinja2 for templating. Create `/ansible/templates/index.html.j2`:

```jinja
<!doctype html>
<html>
<head><title>{{ site_title }}</title></head>
<body>
  <h1>{{ site_title }} — {{ inventory_hostname }}</h1>
  <p>This server is running {{ ansible_distribution }} {{ ansible_distribution_version }}.</p>
  <p>Kernel: {{ ansible_kernel }}</p>
  <p>IPv4 on eth0: {{ ansible_default_ipv4.address }}</p>
  <p>Contact: {{ site_admin_email }}</p>
</body>
</html>
```

Anything `{{ ansible_* }}` is a **fact** — discovered automatically when `gather_facts: true`.

Update the playbook to use the template. Replace the `copy` task in `01-webserver.yml` with:

```yaml
    - name: Render the site index from a template
      ansible.builtin.template:
        src: templates/index.html.j2
        dest: /var/www/html/index.html
        owner: root
        group: root
        mode: "0644"
```

Re-run:

```bash
ansible-playbook -i inventory.ini 01-webserver.yml
```

And `curl http://localhost:8081` should now show a richer page with each host's facts filled in.

## 9. Handlers — only run when something changes

Right now, if you change the nginx config, you'd want to **reload** nginx — but only if the config actually changed. That's exactly what handlers do.

Create `/ansible/templates/nginx.conf.j2`:

```nginx
worker_processes {{ nginx_worker_processes }};

events {
    worker_connections 1024;
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    sendfile on;
    keepalive_timeout 65;

    server {
        listen 80 default_server;
        server_name _;
        root /var/www/html;
        index index.html;

        location / {
            try_files $uri $uri/ =404;
        }
    }
}
```

Replace the contents of `01-webserver.yml` with:

```yaml
---
- name: Configure a basic nginx web server
  hosts: web
  become: true
  gather_facts: true

  tasks:
    - name: Ensure nginx is installed
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Deploy nginx config
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: "0644"
      notify: Reload nginx

    - name: Render the site index from a template
      ansible.builtin.template:
        src: templates/index.html.j2
        dest: /var/www/html/index.html
        owner: root
        group: root
        mode: "0644"

    - name: Ensure nginx is running
      ansible.builtin.service:
        name: nginx
        state: started

  handlers:
    - name: Reload nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded
```

Run it. First run: the template task is `changed` → the `Reload nginx` handler fires at the end of the play. Second run: nothing changed → handler does **not** fire. That's the value of handlers — restart-only-when-needed is built in.

Try changing `nginx_worker_processes: 2` to `4` in `group_vars/web.yml`, re-run, and watch the handler trigger again.

## 10. Loops and conditionals

Real playbooks loop and branch. Quick examples you can drop into a play.

**Loop over a list of packages:**

```yaml
    - name: Install handy tools
      ansible.builtin.apt:
        name: "{{ item }}"
        state: present
      loop:
        - htop
        - tree
        - jq
```

**Loop over a list of dicts (e.g., create users):**

```yaml
    - name: Create app users
      ansible.builtin.user:
        name: "{{ item.name }}"
        shell: "{{ item.shell }}"
        state: present
      loop:
        - { name: alice, shell: /bin/bash }
        - { name: bob,   shell: /bin/sh }
```

**Conditional execution with `when`:**

```yaml
    - name: Install firewalld only on RedHat-family hosts
      ansible.builtin.package:
        name: firewalld
        state: present
      when: ansible_os_family == "RedHat"
```

In our lab both targets are Ubuntu, so that `when` clause would skip the task on both — useful in mixed-OS inventories.

**Run only on one specific host:**

```yaml
    - name: Place a flag file only on target1
      ansible.builtin.copy:
        dest: /etc/primary.flag
        content: "I am the primary\n"
      when: inventory_hostname == "target1"
```

## 11. Roles — organizing real projects

Once a playbook gets past a few dozen lines, you want **roles**. A role is a directory with a standard layout that bundles tasks, handlers, templates, defaults, and files together — reusable, testable, shareable.

Create a role skeleton from inside `/ansible`:

```bash
mkdir -p roles
ansible-galaxy role init roles/webserver
```

This creates:

```
roles/webserver/
├── defaults/main.yml      # default vars (lowest precedence)
├── files/                 # static files to copy
├── handlers/main.yml      # handlers
├── meta/main.yml          # role metadata, dependencies
├── tasks/main.yml         # the tasks themselves
├── templates/             # Jinja2 templates
├── tests/                 # test playbooks
└── vars/main.yml          # role vars (higher precedence than defaults)
```

Move things into the role:

```bash
mv templates/nginx.conf.j2 roles/webserver/templates/
mv templates/index.html.j2 roles/webserver/templates/
```

Put this in `roles/webserver/defaults/main.yml` (overridable defaults):

```yaml
---
site_title: "Default Site"
site_admin_email: "root@localhost"
nginx_worker_processes: 1
```

Put this in `roles/webserver/tasks/main.yml`:

```yaml
---
- name: Ensure nginx is installed
  ansible.builtin.apt:
    name: nginx
    state: present
    update_cache: yes

- name: Deploy nginx config
  ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: "0644"
  notify: Reload nginx

- name: Render the site index from a template
  ansible.builtin.template:
    src: index.html.j2
    dest: /var/www/html/index.html
    owner: root
    group: root
    mode: "0644"

- name: Ensure nginx is running
  ansible.builtin.service:
    name: nginx
    state: started
```

Note: from inside a role, `template:` looks in the role's `templates/` directory automatically — that's why `src:` is just `nginx.conf.j2`, no path prefix.

Put this in `roles/webserver/handlers/main.yml`:

```yaml
---
- name: Reload nginx
  ansible.builtin.service:
    name: nginx
    state: reloaded
```

Now create a thin top-level playbook that just calls the role. `/ansible/site.yml`:

```yaml
---
- name: Configure web servers
  hosts: web
  become: true
  gather_facts: true
  roles:
    - webserver
```

Your `group_vars/web.yml` from earlier still applies and overrides the role defaults (group vars are higher precedence than role defaults — that's the design).

Run it:

```bash
ansible-playbook -i inventory.ini site.yml
```

Same result as before, but now you have a reusable role you could publish to Ansible Galaxy, share with teammates, or apply to a different inventory.

## 12. Ansible Vault — encrypting secrets

You don't want passwords, API keys, or TLS private keys sitting in plaintext YAML in git. Vault encrypts files (or individual variables) with a passphrase.

Create an encrypted vars file:

```bash
ansible-vault create group_vars/web_secrets.yml
```

You'll be prompted for a password, then dropped into an editor. Put something like:

```yaml
---
db_password: "super-secret-value"
api_token: "abc123xyz"
```

Save and exit. Now look at the file — it's encrypted gibberish on disk. To reference these variables in a playbook or template, just use `{{ db_password }}` as normal; Ansible decrypts at runtime.

To run a playbook that uses vault-encrypted vars:

```bash
ansible-playbook -i inventory.ini site.yml --ask-vault-pass
```

Or store the password in a file (chmod 600, never commit) and use `--vault-password-file ~/.vault_pass`.

Other useful commands:

- `ansible-vault edit group_vars/web_secrets.yml` — edit in place
- `ansible-vault view group_vars/web_secrets.yml` — read-only
- `ansible-vault rekey group_vars/web_secrets.yml` — change the password
- `ansible-vault encrypt_string 'mysecret' --name 'db_password'` — encrypt just one value to paste into a normal YAML file

## 13. Tags, check mode, and diff

Three quality-of-life features you'll use constantly.

**Tags** let you run a subset of tasks. Add `tags:` to tasks:

```yaml
    - name: Deploy nginx config
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Reload nginx
      tags: [config]
```

Then run only tagged tasks:

```bash
ansible-playbook -i inventory.ini site.yml --tags config
```

Or skip them: `--skip-tags config`.

**Check mode** is a dry run — Ansible reports what *would* change without changing anything:

```bash
ansible-playbook -i inventory.ini site.yml --check
```

**Diff mode** shows the actual line-by-line changes that would be made to files:

```bash
ansible-playbook -i inventory.ini site.yml --check --diff
```

Combining `--check --diff` before a real run on production is the closest thing Ansible has to a "preview" — get in the habit.

## 14. Tearing down

When you're done playing, from your host:

```bash
docker compose down          # stop and remove containers, keep images
docker compose down --rmi all -v   # also remove images and volumes (full reset)
```

Files you created under `./ansible/` on your host stay put — they're not in the container's filesystem.

## 15. Cheat sheet

```bash
# --- on the host ---
docker compose up -d --build               # start the lab
docker compose ps                          # check status
docker exec -it ansible-controller bash    # shell into controller
docker compose down                        # stop everything

# --- inside the controller, in /ansible ---

# Connectivity & inventory
ansible-inventory -i inventory.ini --list           # show parsed inventory
ansible -i inventory.ini all -m ping                # ping all hosts
ansible -i inventory.ini web -m setup               # gather facts

# Ad-hoc
ansible -i inventory.ini web -m command -a "uptime"
ansible -i inventory.ini web -m apt -a "name=htop state=present" --become

# Playbooks
ansible-playbook -i inventory.ini site.yml          # run
ansible-playbook -i inventory.ini site.yml --check --diff   # dry run + diff
ansible-playbook -i inventory.ini site.yml --tags config    # only tagged tasks
ansible-playbook -i inventory.ini site.yml -l target1       # limit to one host
ansible-playbook -i inventory.ini site.yml -v               # verbose (-vvv for SSH debug)

# Roles
ansible-galaxy role init roles/myrole               # scaffold a new role
ansible-galaxy role install geerlingguy.nginx       # install from Galaxy

# Vault
ansible-vault create group_vars/secrets.yml
ansible-vault edit   group_vars/secrets.yml
ansible-vault view   group_vars/secrets.yml
ansible-playbook ... --ask-vault-pass
```

### Common gotchas in this specific lab

- **Targets can't be reached from your host on port 22** — only ports 8081/8082 are published. SSH is reachable from the controller container via `lab-net` only. That's intentional; do all Ansible work from inside the controller.
- **`service: enabled: yes` will fail** on the targets because they run `sshd` as PID 1, not systemd. Use `state: started` only.
- **`host_key_checking`** — without `host_key_checking = False` in `ansible.cfg` or pre-populated `known_hosts`, the first connection will hang waiting for `yes/no`. Either pre-scan keys (Section 4a) or disable the check in the config.
- **`/ansible` is bind-mounted from your host's `./ansible/` directory.** Edit files however you like, on either side; just don't be surprised when files persist across `docker compose down`.
